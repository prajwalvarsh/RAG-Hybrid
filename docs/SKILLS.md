# Skills — RAG Hybrid Search

## Skill 1: Chunking

### What it is
Breaking a document into smaller pieces (chunks) before indexing.
Each chunk becomes one searchable unit in the vector store.

### Why it matters
LLMs have context limits. You can't feed a 50-page PDF into a prompt.
You retrieve only the relevant chunks, not the whole document.

### Three strategies we implement

**Fixed-size**
- Split every N tokens (we use 512), overlap by M tokens (we use 50)
- Overlap exists so a sentence split across two chunks isn't lost
- Simple to implement, but ignores sentence and paragraph boundaries

**Structure-aware**
- Split at document structure: headings, sections, paragraphs
- A new `##` heading in markdown = new chunk
- Respects the author's own organization of the content

**Semantic**
- Group sentences by topic similarity using embeddings
- When embedding similarity between consecutive sentences drops
  below a threshold, start a new chunk
- Most expensive but produces the most coherent chunks

### Implementation pattern
Each strategy returns the same output shape:
```python
@dataclass
class Chunk:
    text: str
    doc_id: str
    chunk_index: int
    strategy: str        # "fixed" | "structural" | "semantic"
    metadata: dict       # source, page, heading, etc.
```

## Skill 2: Hybrid Search (Dense + Sparse + RRF)

### What it is
Running two different search methods and combining their results.

### Dense retrieval
- Converts the query into an embedding vector
- Finds chunks whose embeddings are closest (cosine similarity)
- Good at: conceptual matches, synonyms, paraphrasing
- Bad at: exact keywords, product codes, proper nouns

### Sparse retrieval (BM25)
- Classic keyword matching with frequency weighting
- Finds chunks that contain the exact words from the query
- Good at: exact terms, error codes, names, technical jargon
- Bad at: synonyms, conceptual questions

### Why you need both
Query: "What is the capital of Bavaria?"
- Dense finds: chunks about Munich, Germany, Bavarian cities ✓
- Sparse finds: chunks containing the exact word "capital" ✓
- Together: much higher chance of finding the right chunk

### Reciprocal Rank Fusion (RRF)
Combines the two ranked lists into one. Formula:

RRF_score(chunk) = (0.7 × 1/(rank_dense + 60)) + (0.3 × 1/(rank_sparse + 60))

The 60 is a smoothing constant. Higher rank = lower number = higher score.
We weight dense higher (0.7) because semantic understanding matters more
for most queries. Tune this after seeing eval results.

## Skill 3: Reranking

### What it is
A second-pass scoring step after hybrid retrieval.

### Why it exists
RRF gives you the top 20 candidates. But RRF scores are approximate —
they're based on rank position, not deep relevance understanding.
A reranker reads the query AND each chunk together and scores true relevance.

### Implementation pattern

Query + Chunk 1  →  reranker  →  score 0.92

Query + Chunk 2  →  reranker  →  score 0.45

Query + Chunk 3  →  reranker  →  score 0.88

...

Keep top 5

### Tool
We use a cross-encoder model from sentence-transformers:
`cross-encoder/ms-marco-MiniLM-L-6-v2`
Free, runs locally, no API key needed.

## Skill 4: Citation Verification

### What it is
After the LLM generates an answer, we check that every cited chunk
actually supports the claim it's attached to.

### Why it matters
LLMs hallucinate. They can cite [1] for a claim that [1] doesn't support.
Most RAG systems skip this check entirely — this is your differentiator.

### Implementation pattern
1. LLM generates answer with bracketed citations: "Munich is in Bavaria [1]"
2. Extract each claim + its citation number
3. For each pair: ask the LLM "does this chunk support this claim? yes/no"
4. Flag unsupported citations in the response metadata
5. Score = supported_citations / total_citations

## Skill 5: Evaluation with RAGAS

### Metrics we track

| Metric | What it measures |
|---|---|
| Faithfulness | Are all claims in the answer supported by retrieved chunks? |
| Answer relevancy | Does the answer actually address the question? |
| Context precision | Are the retrieved chunks actually relevant? |
| Context recall | Did we retrieve all chunks needed to answer? |

### Golden test set rules
- Minimum 50 question-answer pairs
- Hand-written by you — never generated by an LLM
- Include: factual lookups, multi-hop questions, unanswerable questions
- Stored in eval/golden/ — never modified by code
